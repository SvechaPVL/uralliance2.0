## Пользовательский Ввод

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать пользовательский ввод перед продолжением (если не пуст).

## Описание

1. Запустите `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` из корня репозитория и разберите FEATURE_DIR и список AVAILABLE_DOCS. Все пути должны быть абсолютными. Для одинарных кавычек в аргументах вроде "I'm Groot", используйте синтаксис экранирования: напр. 'I'\''m Groot' (или двойные кавычки если возможно: "I'm Groot").

2. **Проверить статус чеклистов** (если существует FEATURE_DIR/checklists/):
   - Сканируйте все файлы чеклистов в директории checklists/
   - Для каждого чеклиста подсчитайте:
     - Всего элементов: Все строки, соответствующие `- [ ]` или `- [X]` или `- [x]`
     - Завершенные элементы: Строки, соответствующие `- [X]` или `- [x]`
     - Незавершенные элементы: Строки, соответствующие `- [ ]`
   - Создайте таблицу статуса:

     ```text
     | Чеклист | Всего | Завершено | Незавершено | Статус |
     |-----------|-------|-----------|------------|--------|
     | ux.md     | 12    | 12        | 0          | ✓ ПРОЙДЕН |
     | test.md   | 8     | 5         | 3          | ✗ НЕ ПРОЙДЕН |
     | security.md | 6   | 6         | 0          | ✓ ПРОЙДЕН |
     ```

   - Рассчитайте общий статус:
     - **ПРОЙДЕН**: Все чеклисты имеют 0 незавершенных элементов
     - **НЕ ПРОЙДЕН**: Один или более чеклистов имеют незавершенные элементы

   - **Если какой-либо чеклист незавершен**:
     - Отобразите таблицу с количеством незавершенных элементов
     - **ОСТАНОВИТЕСЬ** и спросите: "Некоторые чеклисты незавершены. Хотите продолжить реализацию в любом случае? (да/нет)"
     - Ждите ответа пользователя перед продолжением
     - Если пользователь говорит "нет" или "ждать" или "стоп", прекратите выполнение
     - Если пользователь говорит "да" или "продолжить" или "continue", перейдите к шагу 3

   - **Если все чеклисты завершены**:
     - Отобразите таблицу, показывающую, что все чеклисты пройдены
     - Автоматически перейдите к шагу 3

3. Загрузите и проанализируйте контекст реализации:
   - **ТРЕБУЕТСЯ**: Прочитайте tasks.md для полного списка задач и плана выполнения
   - **ТРЕБУЕТСЯ**: Прочитайте plan.md для технологического стека, архитектуры и структуры файлов
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте data-model.md для сущностей и отношений
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте contracts/ для спецификаций API и требований к тестам
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте research.md для технических решений и ограничений
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте quickstart.md для сценариев интеграции

4. **Проверка Настройки Проекта**:
   - **ТРЕБУЕТСЯ**: Создайте/проверьте файлы игнорирования на основе фактической настройки проекта:

   **Логика Обнаружения и Создания**:
   - Проверьте, успешна ли следующая команда, чтобы определить, является ли репозиторий git репо (создайте/проверьте .gitignore если да):

     ```sh
     git rev-parse --git-dir 2>/dev/null
     ```

   - Проверьте, существует ли Dockerfile\* или Docker в plan.md → создайте/проверьте .dockerignore
   - Проверьте, существует ли .eslintrc* или eslint.config.* → создайте/проверьте .eslintignore
   - Проверьте, существует ли .prettierrc\* → создайте/проверьте .prettierignore
   - Проверьте, существует ли .npmrc или package.json → создайте/проверьте .npmignore (если публикуете)
   - Проверьте, существуют ли файлы terraform (\*.tf) → создайте/проверьте .terraformignore
   - Проверьте, нужен ли .helmignore (присутствуют helm чарты) → создайте/проверьте .helmignore

   **Если файл игнорирования уже существует**: Проверьте, содержит ли он существенные шаблоны, добавьте только отсутствующие критические шаблоны
   **Если файл игнорирования отсутствует**: Создайте с полным набором шаблонов для обнаруженной технологии

   **Общие Шаблоны по Технологии** (из технологического стека plan.md):
   - **Node.js/JavaScript/TypeScript**: `node_modules/`, `dist/`, `build/`, `*.log`, `.env*`
   - **Python**: `__pycache__/`, `*.pyc`, `.venv/`, `venv/`, `dist/`, `*.egg-info/`
   - **Java**: `target/`, `*.class`, `*.jar`, `.gradle/`, `build/`
   - **C#/.NET**: `bin/`, `obj/`, `*.user`, `*.suo`, `packages/`
   - **Go**: `*.exe`, `*.test`, `vendor/`, `*.out`
   - **Ruby**: `.bundle/`, `log/`, `tmp/`, `*.gem`, `vendor/bundle/`
   - **PHP**: `vendor/`, `*.log`, `*.cache`, `*.env`
   - **Rust**: `target/`, `debug/`, `release/`, `*.rs.bk`, `*.rlib`, `*.prof*`, `.idea/`, `*.log`, `.env*`
   - **Kotlin**: `build/`, `out/`, `.gradle/`, `.idea/`, `*.class`, `*.jar`, `*.iml`, `*.log`, `.env*`
   - **C++**: `build/`, `bin/`, `obj/`, `out/`, `*.o`, `*.so`, `*.a`, `*.exe`, `*.dll`, `.idea/`, `*.log`, `.env*`
   - **C**: `build/`, `bin/`, `obj/`, `out/`, `*.o`, `*.a`, `*.so`, `*.exe`, `Makefile`, `config.log`, `.idea/`, `*.log`, `.env*`
   - **Swift**: `.build/`, `DerivedData/`, `*.swiftpm/`, `Packages/`
   - **R**: `.Rproj.user/`, `.Rhistory`, `.RData`, `.Ruserdata`, `*.Rproj`, `packrat/`, `renv/`
   - **Универсальные**: `.DS_Store`, `Thumbs.db`, `*.tmp`, `*.swp`, `.vscode/`, `.idea/`

   **Специфичные для Инструмента Шаблоны**:
   - **Docker**: `node_modules/`, `.git/`, `Dockerfile*`, `.dockerignore`, `*.log*`, `.env*`, `coverage/`
   - **ESLint**: `node_modules/`, `dist/`, `build/`, `coverage/`, `*.min.js`
   - **Prettier**: `node_modules/`, `dist/`, `build/`, `coverage/`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`
   - **Terraform**: `.terraform/`, `*.tfstate*`, `*.tfvars`, `.terraform.lock.hcl`
   - **Kubernetes/k8s**: `*.secret.yaml`, `secrets/`, `.kube/`, `kubeconfig*`, `*.key`, `*.crt`

5. Разберите структуру tasks.md и извлеките:
   - **Фазы задач**: Настройка, Тесты, Ядро, Интеграция, Доработка
   - **Зависимости задач**: Правила последовательного vs параллельного выполнения
   - **Детали задач**: ID, описание, пути к файлам, маркеры параллелизма [P]
   - **Поток выполнения**: Порядок и требования зависимостей

6. Выполните реализацию, следуя плану задач:
   - **Выполнение по фазам**: Завершите каждую фазу перед переходом к следующей
   - **Уважайте зависимости**: Запускайте последовательные задачи по порядку, параллельные задачи [P] могут запускаться вместе
   - **Следуйте подходу TDD**: Выполняйте тестовые задачи перед соответствующими задачами реализации
   - **Координация на основе файлов**: Задачи, влияющие на одни и те же файлы, должны запускаться последовательно
   - **Контрольные точки валидации**: Проверяйте завершение каждой фазы перед продолжением

7. Правила выполнения реализации:
   - **Сначала Настройка**: Инициализируйте структуру проекта, зависимости, конфигурацию
   - **Тесты перед кодом**: Если нужно написать тесты для контрактов, сущностей и сценариев интеграции
   - **Основная разработка**: Реализуйте модели, сервисы, CLI команды, конечные точки
   - **Интеграционная работа**: Подключения к базе данных, middleware, логирование, внешние сервисы
   - **Доработка и валидация**: Модульные тесты, оптимизация производительности, документация

8. Отслеживание прогресса и обработка ошибок:
   - Сообщайте о прогрессе после каждой завершенной задачи
   - Прекратите выполнение, если любая непараллельная задача провалится
   - Для параллельных задач [P], продолжайте с успешными задачами, сообщайте о провалившихся
   - Предоставляйте четкие сообщения об ошибках с контекстом для отладки
   - Предлагайте следующие шаги, если реализация не может продолжиться
   - **ВАЖНО** Для завершенных задач, убедитесь, что отметили задачу как [X] в файле задач.

9. Валидация завершения:
   - Проверьте, что все требуемые задачи завершены
   - Проверьте, что реализованные функции соответствуют исходной спецификации
   - Проверьте, что тесты проходят и покрытие соответствует требованиям
   - Подтвердите, что реализация следует техническому плану
   - Сообщите финальный статус со сводкой завершенной работы

Примечание: Эта команда предполагает, что существует полная разбивка задач в tasks.md. Если задачи неполны или отсутствуют, предложите сначала запустить `/speckit.tasks` для регенерации списка задач.
